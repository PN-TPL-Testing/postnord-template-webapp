## CI/CD

| Workflow | Trigger | What it does |
|----------|---------|--------------|
| `ci.yml` | Push to `main` | Builds the Docker image, tags with git SHA, pushes to ECR |
| `deploy.yml` | `ci.yml` succeeds | Registers a new ECS task definition and deploys to the dev service |
| `promote.yml` | Manual (`workflow_dispatch`) | Promotes a given image tag to `tst` or `prd` — Phase 1 stub, currently a no-op |

Every push to `main` automatically builds and deploys to dev. Promotion to tst and prd is always manual.

## Project layout

This app has two parts:

- **Backend** — FastAPI (`app/main.py`), serves the API and the compiled frontend.
- **Frontend** — a React + TypeScript single-page app (`frontend/`) built with Vite, using Azure MSAL for auth. `npm run build` compiles it into `app/static/`, which FastAPI then serves.

## Running locally

Build the React frontend first (required for the SPA to serve correctly):

```bash
cd frontend && npm install && npm run build && cd ..
```

Then start the Python backend:

```bash
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8080
```

Routes are prefixed with `/<APP_SLUG>-<APP_ENV>` (defaults to `/app-dev` when env vars are not set):

| Route | Response |
|-------|----------|
| `GET /<slug>-<env>/` | React SPA (Microsoft Entra auth) |
| `GET /<slug>-<env>/health` | `{"status": "ok"}` |

For frontend-only development, `npm run dev` in the `frontend/` directory runs the Vite dev server with hot reload (port 5173). It proxies API calls to the Python backend if you have that running too.

`APP_SLUG` and `APP_ENV` are injected by ECS at runtime and default to `"app"` and `"dev"` locally.

## Runtime configuration

| Variable | Source | Description |
|----------|--------|-------------|
| `APP_SLUG` | ECS task definition | App identifier, set by the platform at provision time |
| `APP_ENV` | ECS task definition | Environment name (`dev`, `tst`, or `prd`) |
| `ENTRA_CLIENT_ID` | Secrets Manager | Azure AD application (client) ID for MSAL auth |
| `ENTRA_TENANT_ID` | Secrets Manager | Azure AD tenant ID for MSAL auth |

`APP_SLUG` and `APP_ENV` are managed by the platform — do not set them manually in ECS. `ENTRA_CLIENT_ID` and `ENTRA_TENANT_ID` should be added to the Secrets Manager path for each environment (`/apps/<slug>/<env>/config`) and read from there at startup.

## App secrets

Runtime secrets are stored in AWS Secrets Manager at `/apps/<slug>/<env>/config`. The ECS task role for each environment has read access to its own path. Update secrets via the AWS console or CLI — ECS picks them up on the next deployment.

## Promoting to tst or prd

tst and prd ECS services start at 0 replicas. To promote:

1. Go to Actions → **Promote — manual environment promotion**
2. Select the environment and provide the image tag (git SHA from a successful `ci.yml` run)
3. Scale the ECS service up to 1 via the portal or AWS console (Phase 2 of the promote workflow will automate this)
